# v0.1.0 Release Readiness — Design

Date: 2026-07-06
Status: Approved
Goal: Make the public repository (github.com/svcHeidi/4Dpaper) presentable and tag v0.1.0.

## Context

Much of the release groundwork already exists: README with Docker quick start,
LICENSE, `VERSION` at 0.1.0, CI workflows (`docker-publish.yml`,
`pr-regressions.yml`), a hardened prod Docker bundle, and a ~30-file pytest
suite. Remaining gaps found during exploration:

- Untracked `examples/heart` entries need a commit-or-ignore decision:
  `data/` is 63 GB of local simulation data; `_extensions`, `scripts`,
  `_quarto-apphtml.yml`, `_quarto-paperview.yml` are runtime-created symlinks
  to `/app/...` container paths; `sections/` and `.gitignore` are real content.
- Git history carries ~250 MB of accidentally committed build artifacts
  (96 MB and 75 MB standalone HTML exports, a 21 MB compiled report, dozens of
  ~7 MB `.cache/decimation/*.vtk` files). They are deleted and gitignored on
  HEAD, but every clone still downloads them (pack ≈ 80 MB).
- The README instructs users to try `FOURD_WORKSPACE=./examples/heart`, but a
  fresh clone has no `examples/heart/data` (it references a 63 GB
  `monodomainHeart` case plus a Niederer `.foam`), so the full demo cannot
  work out of the box.
- No secrets are tracked (only `.env.example` files and a key-generator
  script) — confirmed by scan.

## Example data resolution (validated locally)

Rather than document the data gap, ship a small committed dataset so the
heart example renders fully from a clean clone. Validated by actually
exporting it:

- Source: the Niederer et al. 2011 benchmark **slab** case (52 MB OpenFOAM,
  8 timesteps). The slab is a synthetic benchmark geometry, cleared for
  public publication.
- Its extracted surface is only 11,050 faces / 11,052 points — far below the
  150k auto-decimation threshold, so **no decimation is needed**; the surface
  is exported as-is.
- All 8 timesteps export to a series totalling **3.5 MB**, preserving all
  five scalar fields (`Vm`, `activationTime`, `ionicCurrent`, `Jsi`,
  `externalStimulusCurrent`). This keeps the ▶ play button, live field
  switcher, and colored maps working offline. 3.5 MB is comfortably
  git-friendly.
- **Format-detection constraint (verified against `scripts/data_loader.py`):**
  `_detect_format` only recognizes the series format when the filename ends
  in `.vtk.series` — a `.vtp.series` name would fall through to the suffix
  map and raise `ValueError`. `load_vtk_series` itself calls `pv.read()` on
  each entry, which handles `.vtp` frames fine. So the index file must be
  named `niederer.vtk.series` even though its per-frame files are `.vtp`.

## Scope

Release-readiness pass, including a post-tag smoke test of the published
ghcr image (added after review — the README already advertises pulling that
image, so it's the actual artifact users will run). Out of scope: CHANGELOG,
CONTRIBUTING, issue templates, demo GIF.

## Plan of Record

### 1. Ship the committed Niederer surface + tidy the example workspace

- Generate `examples/heart/data/niederer/niederer.vtk.series` + 8 `.vtp`
  frames (3.5 MB) from the Niederer slab case using the `SimulationData`
  loader (extract surface, save each timestep, write the series index with
  the `.vtk.series` name per the format-detection constraint above).
- Rewrite `examples/heart/lightweight.qmd` (and trim `main.qmd`) to reference
  the committed `niederer.vtk.series` instead of the 63 GB `monodomainHeart`
  case and the unavailable Purkinje series, so the example renders fully
  offline. Keep the committed `media/` graph + GIFs.
- Add `examples/heart/data/niederer/README.md` documenting provenance: source
  (Niederer et al. 2011 benchmark slab, citation/DOI), the exact export
  script/command used to derive the committed `.vtp` frames from the
  original OpenFOAM case, and redistribution terms under this repo's
  license. This is a public-release gate, not just an attribution nicety —
  confirm before committing that the derived surface data is fine to publish
  under the repo license.
- Extend `examples/heart/.gitignore` to ignore the large `data/monodomainHeart`
  tree and the four runtime-created symlinks (`_extensions`, `scripts`,
  `_quarto-apphtml.yml`, `_quarto-paperview.yml`), while allowing the small
  committed `data/niederer/` series.
- Note: since the Niederer `.foam` source case lives outside this repo (in a
  separate local tree), the export script that generates
  `data/niederer/niederer.vtk.series` cannot be re-run from a clean clone. It
  is a one-time generation step; the committed output is what ships. Keep the
  generation script/command captured in `data/niederer/README.md` (see
  above) for reproducibility, not as a repo-runnable build step.
- Commit the new dataset, rewritten QMDs, `.gitignore`, and
  `examples/heart/sections/`.
- Root-level clutter (recordings, pptx, scratch scripts) is already
  gitignored and untracked — leave untouched.

### 2. Verify tests

- Run the full pytest suite locally. Run the e2e subset only if it works
  without special infrastructure.
- Fix release-blocking failures. Report (do not fix) anything non-blocking.

### 3. Verify the fresh-clone experience

- Fresh clone into a temp dir; `cp .env.example .env`; `docker compose up`.
- Check `/api/health`, the UI loads, and the starter-project path works when
  the workspace is empty.
- Verify the committed heart example renders end to end from the clean clone:
  `FOURD_WORKSPACE=./examples/heart docker compose up`, then explicitly
  render it (e.g. `docker compose exec 4dpapers quarto render
  lightweight.qmd --to html`, or the equivalent `/api/compile` call) and
  confirm `state/figures/` contains the Niederer figure HTML/PNG and that
  the browser shows the play button and field switch. Starting the compose
  stack alone does not prove rendering works — the render step must
  actually run and its output must be checked.

### 4. Docs audit

- Align README format-support claims with the verification-status policy in
  CLAUDE.md §6.0: `.exo`/`.e`/`.ex2`, `.case`, `.cgns`, and real Abaqus
  `.inp` are implemented but not fixture-verified; `.foam`/`.openfoam` are
  externally validated without redistributable fixtures. Do not present these
  as fully verified v1 workflows.
- Fix anything the fresh-clone test contradicts.

### 5. Purge git history

Runs after all content commits land (rewrite changes all hashes):

- Create a backup bundle of the current repo state (`git bundle create`) AND
  push a backup branch to origin (e.g. `backup/pre-filter-repo-<date>`)
  pointing at current `main`, so the pre-rewrite history survives even if the
  local bundle/clone is lost. Do not delete this branch after the rewrite —
  it's the rollback path.
- Dry-run first: list the exact large-blob paths to strip using
  `git rev-list --objects --all | git cat-file --batch-check` filtered to
  blobs > 5 MB, and review the list before running filter-repo — don't rely
  on a fuzzy "any other artifact" match at execution time.
- Run `git filter-repo` against that explicit path list:
  `example_heart-standalone*.html`, `_output/`, `.cache/`, `.quarto/`, plus
  whatever else the dry-run surfaced. The new 3.5 MB
  `examples/heart/data/niederer/` series is committed content, not an
  artifact — confirm it is NOT in the strip list.
- Restore the `origin` remote (filter-repo removes it).
- Post-rewrite verification, before force-push: `git count-objects -vH` to
  confirm pack size dropped, `git fsck` to confirm repo integrity, re-run the
  large-blob scan to confirm the target files are gone, verify `VERSION` and
  `examples/heart/data/niederer/niederer.vtk.series` are still present on
  disk at HEAD.
- Force-push `main`.
- Expected result: clone size drops from ~80 MB to a few MB.

### 6. Tag and release

- Tag `v0.1.0` on the rewritten head (matches `VERSION`).
- Push the tag — triggers the existing `docker-publish.yml` workflow.
- Wait for the workflow to complete, then pull and smoke-test the published
  artifact: `docker pull ghcr.io/svcheidi/4dpaper:0.1.0` and `:latest`, run
  each, confirm `/api/health` returns healthy. This is the artifact the
  README tells users to run (`IMAGE=ghcr.io/svcheidi/4dpaper:latest docker
  compose up`), so it must be verified before calling the release done.
- Create the GitHub release with notes: feature summary, supported-format
  table with honest verification status, Docker quick start.

## Ordering constraints

1. All commits (steps 1–4) land before the history rewrite (step 5).
2. The tag (step 6) is created on the rewritten head so it survives the
   force-push.

## Risks

- Force-push invalidates existing clones; acceptable now (no external
  contributors). Backup bundle + pushed `backup/pre-filter-repo-<date>`
  branch on origin provide rollback even if the local machine is lost.
- Docker verification requires a working local Docker daemon; if the ARM64
  build is slow, verification may take a while but is not blocked.
