Date: 2026-07-06

Goal: Make the public repository (github.com/svcHeidi/4Dpaper) presentable and tag v0.1.0.

## Context

Much of the release groundwork already exists: README with Docker quick start,
LICENSE, `VERSION` at 0.1.0, CI workflows (`docker-publish.yml`,
`pr-regressions.yml`), a hardened prod Docker bundle, and a ~30-file pytest
suite. Remaining gaps found during exploration:

- Untracked `examples/heart` entries need a commit-or-ignore decision:
  `data/` is 63 GB of local simulation data; `_extensions`, `scripts`,
  `_quarto-apphtml.yml`, `_quarto-paperview.yml` are runtime-created symlinks
  to `/app/...` container paths; `sections/` and `.gitignore` are real content.
- Git history carries ~250 MB of accidentally committed build artifacts
  (96 MB and 75 MB standalone HTML exports, a 21 MB compiled report, dozens of
  ~7 MB `.cache/decimation/*.vtk` files). They are deleted and gitignored on
  HEAD, but every clone still downloads them (pack Ōēł 80 MB).
- The README instructs users to try `FOURD_WORKSPACE=./examples/heart`, but a
  fresh clone has no `examples/heart/data`, so the full demo cannot work
  out of the box.
- No secrets are tracked (only `.env.example` files and a key-generator
  script) ŌĆö confirmed by scan.

## Scope

Release-readiness pass. Out of scope: CHANGELOG, CONTRIBUTING, issue
templates, demo GIF, verified public image pull (CI handles publishing).

## Plan of Record

### 1. Tidy the example workspace

- Extend `examples/heart/.gitignore` to ignore `data/` and the four
  runtime-created symlinks (`_extensions`, `scripts`, `_quarto-apphtml.yml`,
  `_quarto-paperview.yml`).
- Commit `examples/heart/.gitignore` and `examples/heart/sections/`.
- Root-level clutter (recordings, pptx, scratch scripts) is already
  gitignored and untracked ŌĆö leave untouched.

### 2. Verify tests

- Run the full pytest suite locally. Run the e2e subset only if it works
  without special infrastructure.
- Fix release-blocking failures. Report (do not fix) anything non-blocking.

### 3. Verify the fresh-clone experience

- Fresh clone into a temp dir; `cp .env.example .env`; `docker compose up`.
- Check `/api/health`, the UI loads, and the starter-project path works when
  the workspace is empty.
- Resolve the example-data gap in the README: state which parts of the heart
  example work from a clean clone (e.g. `lightweight.qmd`, if its data is
  available) and which require external data.

### 4. Docs audit

- Align README format-support claims with the verification-status policy in
  CLAUDE.md ┬¦6.0: `.exo`/`.e`/`.ex2`, `.case`, `.cgns`, and real Abaqus
  `.inp` are implemented but not fixture-verified; `.foam`/`.openfoam` are
  externally validated without redistributable fixtures. Do not present these
  as fully verified v1 workflows.
- Fix anything the fresh-clone test contradicts.

### 5. Purge git history

Runs after all content commits land (rewrite changes all hashes):

- Create a backup bundle of the current repo state.
- Run `git filter-repo` stripping known junk paths:
  `example_heart-standalone*.html`, `_output/`, `.cache/`, `.quarto/`, and
  any other build-artifact blobs > 5 MB.
- Restore the `origin` remote (filter-repo removes it) and force-push `main`.
- Expected result: clone size drops from ~80 MB to a few MB.

### 6. Tag and release

- Tag `v0.1.0` on the rewritten head (matches `VERSION`).
- Create the GitHub release with notes: feature summary, supported-format
  table with honest verification status, Docker quick start.
- Pushing the tag triggers the existing `docker-publish.yml` workflow.

## Ordering constraints

1. All commits (steps 1ŌĆō4) land before the history rewrite (step 5).
2. The tag (step 6) is created on the rewritten head so it survives the
   force-push.

## Risks

- Force-push invalidates existing clones; acceptable now (no external
  contributors). Backup bundle provides rollback.
- Docker verification requires a working local Docker daemon; if the ARM64
  build is slow, verification may take a while but is not blocked.
