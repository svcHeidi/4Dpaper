# 4Dpapers v0.1 LinkedIn Launch Plan

Last revised: 2026-07-13

## Channel decision

Publish from Simão Nieto de Castro’s personal profile. Do not create a second
personal profile for the app. A lightweight Company Page can be added later when
there is a working domain, recurring product updates, recruiting, or paid media.

Before posting:

- Add “Founder and developer of 4Dpapers” to Experience.
- Add the demo and repository to Featured.
- Use a 4Dpapers profile banner.

## Audience and goal

Primary audience: computational researchers and CFD/FEA engineers who publish
3D or time-varying results.

Primary conversion:

> “I want to try this on my dataset.”

Track meaningful replies, datasets offered, successful installs, calls booked,
and repeat users. Impressions are secondary.

## Native demo asset

Create a captioned 15–30 second MP4:

1. Start on a static-looking scientific figure.
2. Rotate it.
3. Play timesteps.
4. Switch a scalar field.
5. Pull back to show that the figure is inside a paper.

Use a readable poster frame. Keep the live demo link visible in the post and in
the first comment; prioritize user access over unverified algorithm folklore.

## Launch post draft

> A 3D, time-varying simulation often becomes one flat screenshot when the
> paper is published. The geometry, fields and timesteps disappear.
>
> I built **4Dpapers** to keep those figures alive.
>
> Write the paper in Quarto Markdown, add supported simulation data, and readers
> can rotate the mesh, play through time and switch scalar fields in the browser.
> The same source also exports a static PDF for conventional publication.
>
> v0.1 is source-available and runs locally with Docker. Its strongest local test
> coverage includes VTK, XDMF/HDF5, common geometry formats and Plotly;
> OpenFOAM has manual validation.
>
> I’m looking for researchers and engineers willing to try it on real data and
> tell me where the workflow breaks.
>
> Live demo: https://4dpapers.github.io/4dpapers/
> Repository: https://github.com/4dpapers/4dpapers

Suggested tags:

    #ScientificComputing #CFD #FEA #OpenFOAM #DataVisualization

## First comment

> Run the bundled example:
>
> git clone https://github.com/4dpapers/4dpapers.git
>
> cd 4dpapers && cp .env.example .env
>
> IMAGE=ghcr.io/4dpapers/4dpapers:0.1.1 FOURD_WORKSPACE=./examples/niederer docker compose up
>
> If you try your own data, tell me the format and what failed. That feedback is
> exactly what v0.1 is for.

## Outreach

- Send personal notes to 5–10 relevant researchers before the public post.
- Ask for a reaction or dataset trial, not a generic reshare.
- Move interested people to one concrete next step: dataset format, Discussion
  thread, or a short call.
- Prepare a private design-partner fallback if the public install path fails.

## Follow-up cadence

- Launch day: product demo and feedback request.
- T+3–5d: technical post about the Quarto/VTK workflow.
- T+7–10d: a real dataset lesson or known limitation.
- T+14d: what early users changed in the roadmap.

The launch is the start of a conversation, not a single irreversible post.
